# Production server (process manager) for both the backend and the enclave.
# The dev server in app.run() (Werkzeug) is single-process and stalls on large
# responses under the TDX CVM; gunicorn gives worker timeouts + better
# write-out. The backend runs multi-worker (`-w N` via FEEDLING_BACKEND_WORKERS
# in gunicorn_conf.py): the :9998 WebSocket ingest is bound by a single
# advisory-lock-elected worker (core/leader.py) and cross-worker state stays
# consistent via Postgres LISTEN/NOTIFY (core/wake_bus.py), so workers no longer
# collide — the old "MUST stay single-worker" constraint is retired. Under the
# FastAPI/ASGI backend gunicorn manages UvicornWorker processes (see the ASGI
# deps below); under legacy Flask it serves the WSGI app directly. The enclave
# embeds gunicorn programmatically (BaseApplication) from enclave_app.py's
# __main__ so its compose entrypoint stays `python -u backend/enclave_app.py`
# (compose_hash unchanged); it serves its in-enclave TLS via the gunicorn
# `ssl_context` hook (needs >=21.0).
gunicorn>=23.0.0

# ASGI/async backend stack (backend/asgi_app.py). The backend is migrating from
# Flask/WSGI to a native FastAPI/ASGI app so waiting I/O (long-poll is the
# bullseye) no longer pins an OS thread. gunicorn runs the app under
# `-k uvicorn_worker.UvicornWorker` (the standalone uvicorn-worker package —
# uvicorn.workers is deprecated since uvicorn 0.30). anyio backs the bounded
# sync->thread bridge (asgi/threadpool.py). No WSGI adapter is pulled in: there
# is no runtime Flask fallback and the main backend imports no flask (only the
# enclave still does — see the flask note at the top of this file).
