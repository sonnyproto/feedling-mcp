## 0. 先理清:记忆有 4 个独立的"轴"(之前全混在一起)

谈记忆一乱,就是因为把下面 4 件不同的事当成一件:

| 轴 | 问的是 | 取值 |
|---|---|---|
| **A 时间尺度** | 这条信息活多久 | 短期(此刻上下文)/ 长期(持久记忆) |
| **B 取用方式**(仅长期) | 怎么把它拿出来用 | 常驻(每轮都在)/ 召回(按相关性才取) |
| **C 种类** | 它是什么性质 | 事实(发生过/为真)/ 推理(agent 的判断,即"TA在想") |
| **D 敏感度** | 隐私属性 + 用户控制 | 普通 / 敏感(+ 用户 看/标/藏/删) |

**一条记忆 = 在这 4 个轴上各占一个位置。** 例:"用户养了狗叫蛋子" = 长期 · (该)常驻 · 事实 · 普通。
**这 4 轴基本正交,应该分开设计,别再缠。** 下面每一块对应一个或几个轴。

## 1. 块一(Seven 这块):格式 / 种类 / type / tab / index —— 轴 C

### 1.1 现状(已全仓核对)
- **捕获**:后端只有**一个统一的 capture worker prompt**(`model_api_runtime/prompts.py:94`),不是每类型一个。每 ~24 轮跑一次(`hosted/turn.py`),temperature=0.1,一次最多 4 条。
- 它**只产 4 个 type**:`fact | event | quote | moment`——而这四个**本质都是"事实"**(quote=带原话的事实,moment=关系性事实,event=带时间的事实,fact=普通事实)。
- **存储 schema 早就共用一套**(`memory/actions.py:_memory_inner_from_action`),所有 type 同一个 envelope。"每条带特定表述"不是 prompt 分叉,是**字段比当初约定的多**(多了 `her_quote/context/summary/verbatim/follow_up/linked_dimension`,其中 `summary` 基本是 `description` 的重复)。
- **关键事实:"TA在想"(insight/reflection)根本没有生成器。** 全仓搜过,capture worker 只产那 4 类;insight/reflection 只能靠带 anchor 的 `/v1/memory/add` 手写,**没有任何自动路径在调它** → **这个板块实际是空的**。schema + 校验(insight ≥1 anchor、reflection ≥2 + 冷却)写了,但上游没人喂。
- **tab**:`TAB_FOR_TYPE` 把 6 type 映射到 3 tab(故事 / 关于我 / TA在想)。
- **index 的痛**:把"事实"和(本应有的)"推理"混在一个 index 里,query 没法统一匹配 → Seven 说"index 没用"的根。

### 1.2 我们的提案(做减法,不是加法;写入路径基本不动)
1. **一种格式**:沿用已共用的 envelope;**砍冗余**(`summary`≈`description`),**留有用可选字段**(`verbatim` 原话 / `context` / `follow_up` / `linked_dimension`)。
2. **现实里只有一种种类:事实。** 把 `fact/event/quote/moment` 从"type(种类)"**降级成 facet/tag(标签)**,挂在那一种格式上,只用于 tab 路由 / 渲染。capture worker 照旧产,**改的是"怎么归类",不是写入逻辑**。
3. **insight/reflection 作为 type 退役** —— 已被现实证明是死的(schema 有、喂的没有)。
   - **"TA在想"若要有内容,新建成"画像"**:一份**从事实增量刷新出来的小结**(生成 + 重写的视图),**不是 anchor 卡、不进事实 index**。
   - **这是新功能(要写生成器)。v1 可以先不做 TA在想 / 先空着**,别再摆空壳。(做不做见 §1.4 决策点。)
4. **tab = 读法/视图**,不是存储种类:
   - 故事 = 关系 moment + 原话(facet 过滤);关于我 = 关于用户的事实;TA在想 = 不做 / 或生成的画像。
5. **index 只盖事实** → 干净、能用;以后叠 embedding。
6. **敏感 = 字段属性 + 用户控制**(见块三)。

### 1.3 场景
- "我养了狗叫蛋子" → 一条事实(facet: fact/关系);
- "蛋子是比熊" → `patch` 补充同一条;
- "其实养的是猫不是狗" → `supersede`(旧卡软退场 + 链新卡);
- "(随口)今天好累啊" → 临时,**不写**(轴 A=短期,见块二);
- "我看你最近压力挺大" → 这是 **agent 的推理(TA在想)**,不是事实 → 进画像层(若做),不进事实库。

### 1.4 ⛳ 决策点(请 Seven / 你拍)
- **D1**:"TA在想 / 画像"——**(a) v1 先不做**(库只剩干净事实)还是 **(b) 重做成"从事实生成的画像"**?
- **D2**:旧 `fact/event/quote/moment` 降级成 facet,前端 3 tab 怎么重排?(故事/关于我保留;TA在想 看 D1)

## 2. 块二:短期 / 长期 / 常驻 / 召回 —— 轴 A + 轴 B

### 2.1 现状
- **短期记忆(轴A:此刻上下文,push,不进长期库)**:屏幕(screen,现在 `on_mention` 关键词触发)、最近聊天(recent chat)、GPS / 位置、pending(待办/未读)。每轮拼进上下文,**用完即弃、不沉淀**。
- **长期记忆(轴A:持久卡)**:就是 memory 卡(块一那些事实)。
- **取用方式(轴B)**:
  - **常驻**:目前**只有 identity**(人设/身份)是每轮都在。
  - **召回**:**所有 memory 卡都只能按 query 召回**(agent-first + 服务端兜底,已闭环)。
- **缺口**:**没有"常驻核心卡(pinned/core)"这一层。** 有些事实(用户名字、核心关系、关键身份事实)**不该等召回**——它们该像 identity 一样每轮都在,但现在不行,得撞上 query 才被捞出来。

### 2.2 目标
| | 现状 | 目标 |
|---|---|---|
| 短期 | screen 关键词触发,其余 push;不沉淀 | 明确"短期边界 + 过期";有价值的**经 A4 蒸馏进长期**(见 §6.4) |
| 长期·召回 | 已闭环(index→selector→fetch / recall) | 保持;叠 embedding 提质 |
| 长期·常驻 | **只有 identity** | **新增 pinned/core 卡层**:一小撮核心事实每轮常驻,不靠召回 |

> 这就是标准的 **core(常驻)vs recall(召回)** 分层——现在缺了 core 那半。

### 2.3 场景
- 用户此刻在看相机评测(screen)→ **短期**,这轮用,过会过期;
- 用户**三天内反复看同款相机**(screen 跨时间)→ A4 蒸馏成**长期**"可能想买相机";
- "你叫什么名字 / 我是谁"(identity / 核心事实)→ **常驻**,每轮都在,不该靠召回;
- 用户提到"我妈" → **召回**关于妈妈的长期事实;
- "我女朋友叫小敏"——是核心关系 → 该进 **pinned 常驻**,而不是每次等召回。

### 2.4 ⛳ 决策点
- **D3**:常驻核心卡(pinned)**谁来定哪些 pin**?(用户手动 pin / 系统按重要度自动 pin / 两者结合)
- **D4**:短期记忆的"过期"与"蒸馏进长期"的边界(多久算过期、什么够格沉淀)。

## 3. 块三:隐私 / 控制权 —— 轴 D

### 3.1 现状
- 隐私是 **M1 引入但只搭了一半的休眠功能**:有 `is_sensitive` 标记 + 查目录(index)过滤 + 刚补的 fetch 过滤;**但没有"何时解锁 / 用户控制"**,所有调用方都传 `include_sensitive=false` → **敏感记忆实际=永久隐身、价值≈0**。
- 敏感"定义"靠**写入时模型随手打标**,无统一政策 → 不可靠。
- **v1 决定**:恋爱陪伴 app,默认藏起来无意义 → **先关掉自动门禁**(`MEMORY_SENSITIVE_GATING_ENABLED` 默认 off,敏感当普通记忆;gate 代码休眠保留,可逆)。

### 3.2 方向(把隐私从"自动门禁"改成"用户控制权")
- 恋爱 app 真正需要的不是会乱判的"敏感自动分类器",而是**用户能在花园里 看 / 标 / 藏 / 删 某条记忆**。
- 隐私 = **事实卡的一个属性 + 用户的控制动作**,不是一套自动解锁逻辑。
- 若将来要"敏感"分级:由**用户标**或**少量明确类目**驱动,不靠模型臆断;"该不该解锁"由**用户显式信号**驱动。

### 3.3 场景
- 用户跟 AI 说一件"想说、但不想被随口提起"的事 → 用户能在花园**手动标/藏**;
- 用户想让 AI"忘掉"某事 → 花园**删**(或 supersede);
- (v1)默认不自动判敏感、不自动藏 → 不打扰、不漏判。

### 3.4 ⛳ 决策点
- **D5**:隐私 v2 = "用户控制(看/标/藏/删)"为主,**砍掉**自动敏感分类器?还是保留一个极简自动层?
