## 0. 一个问题

> **记忆到底分几类?各类怎么用?用户怎么控制?**

之前散成 4 个议题,其实是这一个问题的 4 个切面。下面逐面给现状 + 方向 + 开放问题。

---

## 1. 切面 A · 分类:"事实" vs "agent 推理"

- **现状**:6 个 type(moment/quote/fact/event/insight/reflection)挤在一套存储里;其中 **insight/reflection(= "TA 在想")本质不是记忆,是 agent 对用户的推理/建模**(会变、可能错、要 anchor)。
- **CC 主张(待你和 Seven 拍)**:**先分两类**——
  - **事实层**(发生过/为真:关系 moment、关于用户的事实)→ grounded、可检索、有出处。
  - **推理层**(TA 在想)→ 派生、会变、可能错。**它该不该和事实放一个库/一个 index?**(见切面 D 的 index 问题)
- **开放**:推理层(TA 在想)用法 = **(a) 像记忆一条条存+检索(自带 index、和事实分开)** 还是 **(b) 由事实定期总结生成、根本不进 index(更像实时画像)**?——**这一刀是总开关,定了后面基本顺出来。**

## 2. 切面 B · 格式 / type / tab

- **现状**:存储 schema 其实**已共用一套 envelope**;真正不一样的是"种类/含义",不是数据结构。3 tab(story/about_me/ta_thinking)由 `TAB_FOR_TYPE` 映射。
- **Seven 方向**:一个统一格式 + **不同读/总结方式**;type 可推翻;type↔tab 重做;"前端会变,别拿旧数据结构当依据"。
- **主张**:**tab = 对统一库的"读/总结模式",不是存储 type**;type 大瘦身。
- 详见 `IO-memory-格式与tab-议题-给codex看.md`(本切面的深稿)。

## 3. 切面 C · 隐私 / 控制权

- **现状**:隐私是 **M1 引入但只搭了一半的休眠功能**——有 `is_sensitive` 标记 + 查目录过滤,但**没有"何时解锁/用户控制"**,所有调用方都传 `include_sensitive=false` → **敏感记忆实际=永久隐身、价值≈0**。
- **v1 决定(用户 2026-06-24)**:恋爱陪伴 app,默认藏起来无意义 → **先关掉敏感门禁**(加总开关 `MEMORY_SENSITIVE_GATING_ENABLED` 默认 off,敏感当普通记忆;gate 代码休眠保留,以后翻开)。
- **开放(放本轮一起拍)**:真要做隐私时——谁判定敏感?分几级?用户怎么标/撤/授权?"现在该不该解锁"由什么信号驱动(用户显式提起?设置授权?)。**隐私解锁 = 控制权的一个具体切面。**

## 4. 切面 D · "敏感/分类"由谁定义 + index 为什么没用

- **现状**:一条记忆是不是敏感、是什么 type,**靠写入时模型随手判断打标**,没有统一政策 → 不可靠、不一致。
- **index 没用的根**:事实 + 推理两种不同的东西混在一个 index 里,query 没法统一匹配。
- **主张**:**分清种类(或给每条标 `kind`)+ 统一格式 → index 才匹配得动**(再叠 embedding 更好)。

## 5. 这一刀怎么切(建议讨论顺序)

1. **先拍切面 A 的总开关**:事实 vs 推理分不分?推理层走 (a) 存检索 还是 (b) 总结生成?
2. 顺势定 **kind / type 瘦身**(切面 D)→ index 随之能用。
3. 再定 **tab = 读模式**(切面 B)。
4. 最后把 **隐私/控制权**(切面 C)作为"kind + 用户控制"的一层挂上去。
