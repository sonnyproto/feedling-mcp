# IO Memory Readside M1 · 隐患与待确认

> 2026-06-21 · 作者:Claude(Opus 4.8) · 基于对 `feat/memory-readside-m1` 代码 + Codex 交接文档的核对
> 用途:记录 readside M1 当前的已知隐患和待团队/zhihao 确认的点,避免后续踩坑。

---

## 0. 一句话

readside M1 把召回拆成 `index → fetch` 的底座造好了,但有几处**设计/范围隐患**要记下来——尤其"这套东西在 route B 其实主要是减负、不提精准",别误以为它本身让召回更准了。

---

## 隐患 1 · top 50 是"查询无关"的,卡多了会漏相关旧卡

- **现状**:`_memory_readside_candidates`(`backend/app.py`)只按**元数据**排序取前 50:`is_open_thread → salience → importance → 时间 → id`。**不接收用户的问题/关键词**。
- **问题**:真正相关但"旧 / 低 salience"的卡,如果排在第 51 名,就进不了候选,enclave/agent 永远看不到 → **召回不到**。
- **对比**:现状老路是 enclave 解密**全部**、匹配全部,反而**没有这个 >50 漏卡问题**。
- **影响**:用户卡片 <50 时不暴露(全进得来);**卡一多就是真问题**。
- **建议**:index 这步要不要加一层"按问题/关键词/向量的**相关性预筛**",而不是纯元数据 top 50。→ **待 zhihao 确认。**

---

## 隐患 2 · "pick(从候选里挑)"这步还没实现,主聊天也没接

- **现状**:M1 只写了 `index`(返回 50 条安全摘要)+ `fetch`(按 id 取正文)两个接口。**"拿到 50 条之后谁来挑命中"这步代码没写**;主聊天也没调这两个接口。
- 本地 e2e 脚本里的"agent 挑"是**测试模拟**,不是线上流程。
- **影响**:现在线上召回走的还是老路;新接口是"造好了、本地验证了,还没接进主管道"。
- **建议**:明确"pick 由谁做、主聊天什么时候切过来"。→ 见隐患 4 的路线分叉。

---

## 隐患 3 · route A 没接、也没收口

- **现状**:Codex 文档明确"没有完整收口 route A";"route A 是否接同一套 index/fetch"在给 zhihao 的问题里**仍待定**。
- **更深的老问题**:route A 的 agent 默认用**自己的记忆**,绕开 IO 卡库——就算 index/fetch 摆在那,也得先让 agent **真去调、真用 IO 记忆**(= route A 收口),才有意义。
- **反直觉**:route A 其实是 agentic index/fetch **最自然的归宿**(见隐患 4),问题不在"不行",在"没接 + 没收口"。
- **建议**:确认 index/fetch 是否对 route A 的 agent 暴露(MCP/HTTP);route A 收口单独立项。

---

## 隐患 4 · 这套架构的价值是"route-dependent"的(最重要)

**别误以为 index/fetch 本身让召回更准了。** 拆开看:

| 路线 | pick 谁做 | 精准度 | 额外成本 | 价值 |
|---|---|---|---|---|
| **route B + 算法 pick** | 复用关键词算法 | **不提升**(和现状一样关键词) | 无额外 LLM | **只减负**(少解密、少塞 token);还白送隐患 1 |
| route B + LLM pick | 服务器模型 | 提升 | **+1 次服务器 LLM** | 提精准但贵 |
| **route A + agent pick** | 用户自己机器上的 agent | **真提升**(语义,不是关键词) | **不占服务器 LLM** | ✅ 真正有意义 |

**结论:**
- **route A:agent 语义挑 = 精准真提升,且免费(用户自己的 agent)。这才是这套架构的价值所在。** 靠 skill 定义让 agent 调两次接口即可。
- **route B:算法 pick 是横向移动**——不提精准,只减负/省 token;真要提精准只能加 embedding(post-M1),或接受 +1 次 LLM。
- 所以 **route B 切不切到 index/fetch 不急**(为精准切没意义,为减负可选);**route A 接入才是重点**。

---

## 隐患 5 · 算法 pick 复用 = 继承关键词局限

- **决定**:算法 pick 应**复用 `context_memory_selection.py`**(实体/短语/稀有词级联已调好、防误命中、处理中英文),**不要重造**关键词匹配器。
- **但要清醒**:复用它 = **精准度和现在一样**(关键词,paraphrase 翻不到)。这正是隐患 4 里"route B 算法 pick 不提精准"的原因。
- **想真提精准**:加 embedding/向量(新建组件),post-M1。

---

## 给 zhihao / 团队的待确认清单

1. **§8(每次别走两次 LLM)是 Codex 的工程结论,不是 zhihao 拍的**——要采纳的话请 zhihao 确认一句。
2. **index 要不要加"相关性预筛"**(隐患 1):纯元数据 top 50 在卡多时会漏相关旧卡。
3. **主聊天什么时候切到 index/fetch**(隐患 2)。
4. **route A 是否接同一套接口 + 何时收口**(隐患 3)。
5. **明确定位**:route B 走 index/fetch 主要是减负(不提精准),团队是否认可这个取舍(隐患 4)。
6. **post-M1 是否规划 embedding** 作为唯一能让 route B 也提精准、又不加 LLM 的路(隐患 5)。

---

## 一句话收口

> **readside M1 底座没问题,但要记住三件事:① top 50 查询无关,卡多会漏(要不要加相关性预筛待定);② pick 和主聊天接入还没做;③ 这套的真价值在 route A(agent 语义挑、免费、提精准),route B 算法 pick 只减负不提精准——别把"减负"当成"召回变准了"。**
