# IO Memory M3:记忆质量 / Eval / Prompt 策略(方向 + 已发现的 bug)

> 2026-06-22 · 整理:Claude(CC) · 状态:方向记录,未开工
> 定性:**M1/M2 是"修管道"(能读/能写/能取代);M3 是"调脑子"(该不该记、记什么、怎么用)。M3 的本质是 eval 驱动的 prompt 调优**,不是加功能。

---

## 0. ⚠️ 已发现的 bug(M3 的起点,别忘)

**现象**(M2 测试中):用户自然聊天里明确说了——
- "我现在养了条狗,就蛋子"
- "蛋子是一只比熊"
- "屁股上有一个胎记"

聊天记录里确实有,但**没有生成 `memory.create`**。

**这不是 M2 存储 bug**(写入链路已验证:"我喜欢在海边"能成功写成 v1 卡)。问题在更前面:**后台 memory controller 没把"自然聊天里陈述的持久事实"判定为该写入。**

**根因(已核对代码 `hosted_runtime.py:build_background_execution_messages` 的 prompt):**
controller 的捕获标准写的是"用户**要求**记 / 显式的第一人称持久**偏好**或纠正"。所以——
- ✅ "我喜欢在海边" = **偏好(preference)** → 命中。
- ❌ "我养了条狗叫蛋子" = **事实(fact)**,既不是偏好、也没说"记住" → **掉出捕获标准**。

→ **结论:controller 偏向捕获"偏好 + 显式要求记",对"自然陈述的持久事实(宠物 / 人 / 地点 / 物品 / 关键生活事实)"捕获不足。**
(补充:就算当场没捕获,还有每 24 轮的 capture 兜底——但短测试跑不到 24 轮,所以当场漏=测试里就漏。要"实时记住狗",必须改当场 controller 的标准,不能靠 24 轮。)

---

## 1. M3 核心目标(5 条链系统化)

1. **recall**:怎么查 index、何时 fetch、没查到别瞎答。
2. **写入判断**:什么该记 / 不该记,何时 create / patch / supersede。 ← **狗的缺口在这**
3. **整理**:多轮补充合并成一张更完整的卡(merge);旧事实退场;弱/冲突/过期记忆处理。
4. **回复 prompt**:有记忆就用、没有就承认不知道、别编;用户当前纠正优先于旧 memory。
5. **eval/probe**:把上面行为变成可重复测试,防 prompt 被改坏。

---

## 2. M3 的命门:写入判断的 precision / recall 取舍

> **捕获放宽(抓到狗)↔ 捕获收紧(不记错 / 不记临时 / 不被幻觉污染)** —— 这是写入判断的两难。

放宽抓到狗,但也可能把"今天想吃火锅""模型自己编的""随口一说"都记进去。**这个平衡点没法拍脑袋,只能靠 eval 量出来。** 所以 eval 不是 M3 的尾巴,是脊柱。

---

## 3. 执行建议(CC)

1. **eval 先行、当脊柱**:先立一组 probe(**该记 / 不该记 / 该 supersede / 该承认不知道**),再动任何 prompt。这正是一直被 defer 的那个 memory eval 的归宿。
2. **先做"写入判断"这条链**:狗的缺口是已证实、最高价值的入口,改动集中在 controller prompt + probe。把捕获标准**扩到 durable facts**(实体类:宠物/人/地点/物品/关键生活事实),不只 preferences。
3. **顺序**:写入判断(+eval)→ 回复 prompt → recall → **整理(merge)排最后**(merge 是 M2 故意推迟的新功能,较重)。
4. **别和已有的重复**:recall 里"没查到别瞎答""用户当前纠正优先于旧 memory",**M1.5 的 fallback bug 修复已做了一部分**(那个"被自己错话带偏"的修复)——接着往上加 probe,别重造。

---

## 4. 一句话收口

> **M3 = 用 eval 调记忆的"判断力"。起点是一个已证实的 bug:自然陈述的持久事实(狗=蛋子=比熊=胎记)没被捕获,因为 controller 偏"偏好"漏"事实"。先立 probe(脊柱)→ 先扩写入判断到 durable facts → 回复/recall → merge 最后。命门是"捕获 precision/recall",只能靠 eval 定。**
