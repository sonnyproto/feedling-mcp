# IO 记忆结构方案(给 Sevenfloor 确认)

> 2026-06-24 · hx / CC 整理 · 给 Seven 拍板
> 一句话:我们把记忆结构**大幅简化**了,方向就是你说的"统一格式 + 不同读法 + type 可推翻 + 前端会变"。这份请你确认是否对路。

---

## 1. 一句话结构

**记忆 = 一种极简卡(不再分 6 种 type);"我们的关系/人设"= identity(单独、用户控制);TA在想(推理)先不做。检索 = agent 自己挑;Garden = 用户的控制台。**

---

## 2. 记忆卡(极简,一种格式)

不再有 fact/event/quote/moment/insight/reflection 六类,**全塌成一种卡**:

| 字段 | 是什么 |
|---|---|
| `kind` | **关系(relationship) / 事实(fact)** —— 只决定"怎么被取出来"(见 §3),**不是显示分类**,不是旧 6 类 |
| `content` | 一段正文(Markdown):记忆 + 当时上下文 + 使用提示(怎么自然用) |
| `summary` | 一句摘要(给检索/列表用) |
| `emotion_weight` | **0–1:这条记忆"未来对理解用户有多重要"**(不是语气多激烈)。住焦作=低,"难过时别给建议"=高。**写入时由模型按档打分**(普通事实 0.1–0.3 / 偏好宠物 0.4–0.6 / 情绪关系边界 0.7–0.85 / 强情绪核心 0.9–1.0) |
| `decay`(派生,不存) | 衰减程度。**不是存的数,是读取排序时当场算**:越久没被用到越高、排得越后;**被再次提到就回升**(更新"上次用到时间");`emotion_weight` 高的衰减更慢。无后台任务 |
| `source` | 出处(聊天 / 感知)—— 决定可信度 + 怎么说 |
| `occurred_at` | 时间 |

→ 这就是你说的"**统一格式 + 不同方式读/总结**"。type 不再是"种类",只剩一个 `kind` 决定取法。

---

## 3. 两层:关系(气氛灯,常带) vs 事实(档案柜,按需查)

**按"行为/取法"分,不按"内容"纠结**("关系/身份/事实"的界限本来模糊):

- **identity = 常驻**(名字/角色/称呼/边界/关系维度/稳定约定)——每轮都在,**用户控制,系统不偷偷改**。
- **关系记忆(kind=relationship)= 气氛灯**:**每轮主动带几条**(最动情的底色 + 当前相关的)——保证**陪伴感/关系味道一直在**,不靠关键词搜(这正是怕"关系被埋"的解)。
- **事实记忆(kind=fact)= 档案柜**:**需要时 agent 自己查**(越来越多,不每轮摊出来);agent 没查时系统兜底查一次。

> 关键:关系**不能只靠搜**(会被埋)→ 所以"常带";事实量大 → "按需查"。两种取法不同,所以保留 `kind`。

**每轮注入构成(数量,可配、靠 eval 调)**:
```
identity              常驻,每轮都在
+ 关系记忆 ambient 2   每轮带:最动情/最近的关系底色(气氛灯,不靠搜)
+ 最近记忆 2(任意类)  每轮带:刚说过/刚确立的,别转头就忘
+ 事实记忆(agent 查)  需要时 agent 自己查,没查则系统兜底补几条
总数封顶 ~10(给关系/最近留位,事实让位;超了砍)
```
(关系"按当前话题再多带几条"= 后置,等语义检索;v1 主要靠 ambient 那 2 条 + 最近 2 条。)

## 4. 我们砍了什么(对应你的方向)

| 砍掉 | 为什么 |
|---|---|
| 6 个 type + 3 个 tab(故事/关于我/TA在想)| type 可推翻;tab 改成"读模式",不是存储分类 |
| **insight/reflection(TA在想)** | 现在**根本没生成器、是空的**——先去掉;以后做成"**推理层/画像**"(见下) |
| 一大堆重字段(title/description/her_quote/verbatim/…) | 都塌进 `content`(MD);它们大半是为旧前端样子存的 |
| supersede/纠错的复杂机器 | 改口=新写一条+靠日期判断;以后真需要再做轻量版 |

→ 全是**做减法**,契合你"别拿旧数据结构当依据、前端会变"。

### 未来:一个"推理层 / 画像"(先解释清楚,v1 不做)

旧的"TA在想"(insight/reflection)本想**存成很多条卡**(一条想法一张卡)——但它没生成器、一直是空的,而且卡会堆积、过时、互相矛盾。

**新做法:不存一堆卡,而是维护"一段会自动刷新的画像"**——就**一段话**,描述"我现在对你的理解",**定期从事实/互动里重新生成更新**。
> "刷新的摘要,不是一堆卡" = **一段不断更新的"对用户的理解"总结,而不是几十张零散的 insight 卡。**

这段画像 = **agent 对用户的猜测**,既从**聊天**猜("难过时要陪"),也从**屏幕/设备行为**猜("看了 3 天相机→可能想买")。**是猜测不是事实**:试探性地用、会变、可推翻,**单独一层、不混进 fact/relationship**(否则猜测会污染事实)。

> **注意:画像 和 "推测/推理" 是同一个东西**(一段刷新的猜测总结),不是两层。

### 现状 + 排期(给你看清楚)
- **现在没有画像/推测**(旧的 insight/reflection 是空壳,已删)。
- **v1 不做它**——**因为要上线了**,先把**扎实的两层(事实 + 关系)**上稳;陪伴感靠 relationship 层已经有,画像是锦上添花的"猜测层",不是地基。
- **画像/推测 = 和 eval 一起做**(下一阶段):因为它是"猜测",**必须有 eval 兜着**(防 agent 一本正经说错对用户的猜测),所以天然和质量 eval 绑在一起、放上线后那一波。

**所以现在的三层 = `identity(有) + memory:fact+relationship(v1 做) + 推理层/画像(没做,和 eval 一起,后续)`。**

---

## 5. Garden = 用户的控制台

既然系统不自动改 identity,**改的人是用户,在 Garden 改**:
- 看 / 删 / 纠正记忆;
- 编辑 identity(人设/关系);
- (以后)标隐私。

Garden **不再按 type 分 tab**,按这个新模型重做(后端模型定 → 前端跟着长)。隐私 v1 先不做自动门禁,用户能删就够。

---

## 6. 请你确认/拍的点

1. **TA在想/画像**:v1 **先不做**;以后做成"从事实自动生成的画像(刷新的摘要)",不是 insight/reflection 卡。—— 认同吗?
2. **一种卡 + 旧 6 type/3 tab 弃用、tab=读模式**;只保留一个 `kind`(关系/事实)决定取法。—— 认同吗?
3. **核心字段 = `emotion_weight`(对理解用户的重要度,不是语气强度)+ decay + source**。—— 符合你想要的"质感"吗?
4. **三档**:identity(用户控制常驻)/ **关系记忆=气氛灯,每轮常带** / 事实记忆=按需查。关系不靠搜、主动带。—— 这个切法你认吗?
5. **Garden = 用户控制台,按新模型重做,隐私 v1 先简化**。—— OK 吗?

确认后,这就是记忆 v1 的结构基线,后端照它实现、前端照它重做。
