# IO 记忆统一架构 · 大白话

> 2026-06-23 · 给非技术/快速理解用。配套:`IO-memory-统一架构-spec-给codex-review.md`(架构)+ `IO-memory-统一架构-plan.md`(5 阶段执行)。

## 1. 问题:两条路像两个 app

用户用 IO 有两种接入方式:
- **API 方式(route B)**:用我们托管的服务,自带模型 key,聊天在我们这跑。
- **VPS 方式(route A)**:用户自己的 agent(CC/Hermes 这种)跑在自己机器上,我们的"中转(consumer)"帮它和 IO 通信。

这两条路的**记忆逻辑各做各的**,差到领导说"像两个不同的 app"。要把它们合成一套。

## 2. 核心想法:一条写、一条读,啥来源都汇进去

打个比方,**中央厨房**:
- 一个**进料口(写)**:不管食材从哪来,都进同一套流程做成菜(记忆卡)。
- 一个**出餐口(读)**:要用的时候,从同一个地方取。
- **食材(来源)**:聊天、录屏、GPS、屏幕感知……都从同一个进料口进。

现在是"两家分店各炒各的",要改成"一个中央厨房"。

## 3. "写记忆"是什么(三步)

1. **判断该不该记**(propose):用户说的话里,哪些值得长期记住?
2. **整理成标准动作**(coerce):把"要记这个"翻译成系统认的格式。
3. **存进去**(commit):写成一张记忆卡。

> 好消息:**第 3 步(存)早就统一了**——两条路最后都走同一个写入口。乱的是前两步。

## 4. "读记忆"是什么(两层)

每次回答用户前,要给模型拼一段"上下文",分两层:
- **短期**:最近几句聊天、当前屏幕、GPS、待确认的事——**用完就扔**。
- **长期**:从记忆库里**召回**回来的卡(比如"用户有只猫叫武松")——**持久的**。

把这两层拼成一段喂给模型,它才答得准。

## 5. 现在到底卡在哪

| 环节 | 状态 |
|---|---|
| 存(写的终点)| ✅ 已统一 |
| 判断该不该记 | ❌ 两条路各写各的,还碎成好几处(**"我养了狗叫蛋子"没被记住**就是这的 bug)|
| 整理成动作 | ❌ 两套,会跑偏 |
| 读取召回 | ❌ **VPS 根本没接**(它只用 agent 自己的记忆)|
| 上下文怎么拼 | ❌ 两条路各拼各的 |
| 录屏/GPS/感知 | ❌ 只用一下就扔,**不进长期记忆** |

## 6. 怎么改:分 5 步

- **P1 对齐**:两条路用同一套"动作格式"、同一个"拼上下文的能力";**VPS 第一次能用上 IO 的长期记忆**。(只对齐,不改判断)
- **P2 统一判断 + 修狗 bug**:把"该不该记"合成一套规则,两条路共用;**把判断从只记"喜好"扩到也记"事实"**(宠物、人、地点),以后"养了狗叫蛋子"能被记住。
- **P3 装质检(eval)**:搭一套测试,系统地量"记得对不对、用得对不对",照着调。
- **P4 接新食材**:让屏幕/GPS/录屏这些**也能沉淀成长期记忆**(比如"反复看某件衣服→可能想买")。等隐私设计和质检成熟再做。
- **P5 换更好的检索灶台(可选)**:如果召回不够好,再离线试试 MemPalace 这个检索库,验证安全了才考虑用。

主线是 P1→P2→P3,P4/P5 后置。

## 7. 几个关键决定(大白话)

- **谁是 agent 谁来判断**:API 是我们服务器判断;VPS 是用户自己的 agent 判断——**不在服务器替 VPS 多跑一个 AI**(那费钱、也违背"用户自带算力")。
- **共享"规则"不共享"代码"**:两条路用同一份判断规则和格式标准,但各自实现(因为它们是两个独立进程,硬塞同一段代码会脆)。靠一组共用测试保证不跑偏。
- **永不硬删**:纠正旧记忆 = 写新卡 + 旧卡退场(划掉归档),旧的留着可追溯。
- **行为变了才测**:只对齐(P1)轻冒烟就行;一改"该记什么"(P2 起)必须先过测试才放量。
- **MemPalace 先不用**:它是检索库、不是测试工具,而且会变成第二套记忆源、和我们的加密边界冲突——只当以后的离线试验。
- **不走 MCP**:VPS 走普通后端接口,不走那条已经弃用的 MCP 路。

## 8. 一句话

> **把"记忆"从两条路各做各的,收成一条写(判断→整理→存)+ 一条读(短期+长期召回);聊天、录屏、GPS 都当食材插到同一套管道上。先让两条路不再像两个 app(P1),再统一判断顺带修狗 bug(P2),然后装质检(P3),最后接感知和考虑换检索引擎(P4/P5)。**
