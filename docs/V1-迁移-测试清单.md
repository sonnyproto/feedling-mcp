# V1 老卡迁移 · 测试清单(单独,最后测)

> 迁移是**后台静默**件:用户进安静窗口 → 后台把老格式记忆卡原地升级成 v1(保 id、不丢内容)。
> 两半:**人工真机观测**(Garden 看老卡变 v1)+ **Codex 可脚本 e2e**(造老卡→打 tick→验不变量)。
> 前置代码:`feat/memory-card-migration`(已 Codex 批准)。

## 0. 加速旋钮(测试用,生产默认不动)

```
FEEDLING_MIGRATE_WINDOW_SEC=60       # 每分钟一批

# 且 /v1/capture/tick 有定时器在打(或手动打)
```

## A. 怎么造老卡(没有老卡测不了迁移)

老卡 = 加密 inner 是 `{title/description/her_quote/context/linked_dimension}`(缺 v1 的 summary/content/bucket/threads)。来源任选:
1. **真·老账号**:上线前 onboard 过、库里本来就是老卡 —— 最真。
2. **DB 直插**(Codex e2e):往 `memory_moments` 插一张 inner 为老形态的加密卡。
3. **老格式 add**:若 `/v1/memory/add` 接受 title/description 形态 inner,可直接造。

## B. Codex 可脚本 e2e(验不变量,推荐先跑这个)

> Codex:对 test 部署或本地后端,脚本化以下,逐条断言。

1. **造 3 张老卡** + 记下它们的 `id` 和原文(如"去西湖"/"狗叫蛋子")。
2. **看初始降级读**:`GET /v1/memory/list` 或 readside → 老卡能读到(summary≈title、bucket="未分类")—— 证明降级不崩。
3. **触发**:`POST /v1/capture/tick`(用户态、安静窗口已过)→ 响应 `migrate.enqueued=true`。
4. **等 consumer 处理一批**(或手动跑 resident consumer 一轮)。
5. **断言(核心不变量)**:
   - ✅ **id 全不变**:迁移后这 3 张还是同样的 id(不是新卡)。
   - ✅ **内容不丢**:原文（西湖/蛋子）仍在升级后的 summary/content 里。
   - ✅ **变 v1 形态**:`legacy_batch` 再查 → 这几张不再是 legacy;bucket 不再是"未分类"占位、threads 有值。
   - ✅ **state 推进**:`GET /v1/memory/migration_state` → `legacy_remaining` 递减,排干后 `status=done`。
6. **并发安全**:迁移前先改其中 1 张(content_patch/supersede)→ 跑迁移 → 断言**你的改动没被冲掉**(upgrade 的 CAS：被改过的那张 `skipped:stale`、留下轮)。
7. **幂等**:再打一次 tick → 已 v1 的不再被当老卡、不重复迁;`status=done` 时不再 enqueue。
8. **token-only(HOST_ALL)**:用 runtime token(无 X-API-Key)走一遍 legacy_batch → 仍能解开老卡(已有回归 `test_memory_legacy_batch_token`)。

## C. 人工真机观测(给 hx)

**前置**:一个有老卡的账号 + 上面的加速旋钮 + tick 在打。

### C1 ⭐ 老卡自动变规整
- **【做什么】** 用有老卡的账号,进 Garden 看一眼现状(老卡可能挤在"未分类"、没线索)→ 然后放着别动、过几个安静窗口。
- **【✅成功】** 过一会儿再看 Garden,老卡**逐渐**归到合理的桶、有了线索,summary/正文更规整;**卡还是那些卡**(没凭空多出来、没消失)。
- **【❌坏】** 卡消失/重复/内容丢了/Garden 崩。

### C2 内容没丢
- **【做什么】** 迁移前记住某张老卡讲的事("我家狗叫蛋子")→ 迁移后问它 / 看那张卡。
- **【✅成功】** 那件事还在、答得出。
- **【❌坏】** 内容变了/丢了/被编造。

### C3 不打扰正常使用
- **【做什么】** 迁移期间正常聊天、加新记忆。
- **【✅成功】** 聊天/新记忆一切正常,不卡、不黑屏;你新加/新改的卡不会被迁移冲掉。
- **【❌坏】** 聊天卡顿/记忆错乱/新写被覆盖。

---

## 红线(任一 ❌ 别上)

- **id 变了 / 卡丢了 / 卡重复**(迁移必须原地、保 id)。
- **内容丢失或被编造**。
- **冲掉用户并发新写/新改**(CAS 必须挡住)。
- 迁移期间**读崩或显示坏卡**。

> 注:迁移是"老卡短暂降级可接受"的低优后台件 —— 即便慢、即便某张这轮没迁到,下个安静窗口会再来,**不阻塞 v1 上线**。红线只在"丢/错/冲"。
