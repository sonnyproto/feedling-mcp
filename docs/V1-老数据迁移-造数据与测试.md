# V1 老数据迁移 · 造数据 + 测试(给 hx)

> 配套脚本:`tools/seed_legacy_memory.py`。完整不变量/红线见 `docs/V1-迁移-测试清单.md`,本篇专讲"怎么造老数据 + 一条龙怎么测"。

---

## 0. 先搞清楚:为什么不能用普通写入造老卡

- **老卡 = 解密后的 inner 是老形态**:`{title, description, her_quote, context, linked_dimension}`,**且没有** v1 的 `{summary, content, bucket, threads}`。判定见 `backend/memory/migration.py:is_legacy_card_inner`。
- **普通 `memory.add` 造不出老卡**:写路径 `/v1/memory/actions` 会过 `_memory_inner_from_action`,**永远归一化成 v1 形态**。所以"用老格式 add 直接造"行不通(旧测试清单 A.3 那条是错的,已订正)。
- **唯一的应用层造法**:把老形态 inner 用 `content_encryption.build_envelope` **自己封一个信封**(和 consumer capture 写卡同一套 seal),再走**预封装信封**的 `memory.add` 写进去。`seed_legacy_memory.py` 就是干这个。

## 1. 造老数据(一行命令)

**在 consumer checkout 里、带 consumer 的 env 跑**(它读的就是 consumer 那套 `FEEDLING_API_URL` / `FEEDLING_API_KEY` / `FEEDLING_ENCLAVE_URL`):

```bash
# 默认造 10 张老卡(去西湖 / 狗叫蛋子 / 怕香菜 / 妈妈住院 …,occurred_at 拉开成一段历史)
python3 tools/seed_legacy_memory.py

# 只造前 3 张(够测就行)
python3 tools/seed_legacy_memory.py --count 3
```

- 默认 10 张 > 一批(`FEEDLING_MIGRATE_BATCH` 默认 8)→ 顺带能测**多批 + remaining 递减**。
- 脚本会打印写入响应。**记下新卡的 id**(或下一步用 `legacy_batch` 读回),迁移后要断言 id 不变。
- 每张卡里塞了好认的原文("狗叫蛋子""怕香菜"),迁移后用来验**内容没丢/没被编造**。

## 2. 测试加速旋钮(测试用,生产别动)

迁移默认安静窗口 1200s、每小时一批,测起来太慢。临时环境变量:

```bash
FEEDLING_CAPTURE_QUIET_SEC=30     # 安静窗口缩到 30s
FEEDLING_MIGRATE_WINDOW_SEC=60    # 每分钟一批
FEEDLING_MIGRATE_BATCH=10         # 一批 10 张(默认 8)
# 另需主流程那套:FEEDLING_GENESIS_WORKER_ENABLED / RUNTIME_TOKEN_SECRET / ENCLAVE_URL
# 且要有定时器在打 /v1/capture/tick(或手动打)
```

## 3. 一条龙(seed → 看老态 → 触发 → 看 v1)

1. **造**:`python3 tools/seed_legacy_memory.py`(记下 ids)。
2. **确认是老态**:`POST /v1/memory/legacy_batch` → 刚造的几张应出现在 batch 里(`is_legacy` 命中)。
   - 同时 `GET /v1/memory/list`/readside → 这几张也能读到(降级显示:summary≈title、bucket≈"未分类")—— 证明老卡降级读不崩。
3. **触发**:安静窗口过了之后 `POST /v1/capture/tick`(用户态)→ 响应里 `migrate.enqueued=true`。
4. **等一批被处理**(或手动跑一轮 resident consumer)。
5. **验**(核心不变量):
   - ✅ **id 不变**:那几张还是原来的 id(原地升级,不是新卡)。
   - ✅ **内容不丢**:原文("去西湖""蛋子""香菜")仍在升级后的 summary/content 里。
   - ✅ **变 v1**:再 `legacy_batch` → 这几张不再是 legacy;bucket 不再是"未分类"占位、threads 有值。
   - ✅ **state 推进**:`GET /v1/memory/migration_state` → `legacy_remaining` 递减,排干后 `status=done`。
6. **并发安全**:迁移前先改其中 1 张(content_patch / supersede)→ 再迁 → 断言**你的改动没被冲掉**(CAS:被改过的那张这轮 `skipped:stale`、留到下轮)。
7. **幂等**:再打一次 tick → 已 v1 的不再被当老卡;`status=done` 时不再 enqueue。

## 4. 真机观测(给 hx,不看接口看 Garden)

- 用造好老卡的账号进 **Garden**,先看现状(老卡可能挤在"未分类"、没线索)→ 放着别动、过几个安静窗口。
- ✅ 再看:老卡**逐渐**归到合理的桶、有了线索、summary/正文更规整;**卡还是那些卡**(没多、没少)。
- ✅ 迁移前记住某张老卡的事("狗叫蛋子")→ 迁移后问它 / 看那张卡,**事还在、答得出**。
- ✅ 迁移期间正常聊天/加新记忆,**不卡、不乱、新写不被冲**。

## 红线(任一 ❌ 别上)

id 变了 / 卡丢了 / 卡重复 / 内容丢失或被编造 / 冲掉用户并发新写 / 迁移期读崩。
(迁移是低优后台件,"某张这轮没迁到"不是红线 —— 下个窗口会再来。)详见 `V1-迁移-测试清单.md`。
